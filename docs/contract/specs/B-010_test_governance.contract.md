# B-010: Core Test Governance (Contract)

## 1. 개요 (Overview)
본 Contract는 `PRD-010: Core 최소 테스트 존재 규칙 및 승인 흐름 고정`에 기반하여, 시스템의 핵심 영역(Core)에 대한 최소한의 테스트 존재 요건과 승인 절차를 명문화한 계약이다.

## 2. 물리적 경계 및 적용 대상 (Scope)

### 2.1 Core 경계 정의
- **대상 경로:** `src/core/*` 디렉토리 하위의 모든 코드.
- **대상 브랜치:** `main` 브랜치에 존재하는 해당 계층 코드.

### 2.2 적용 시점 (Non-Retroactive Rule)
- 본 계약은 PRD-010 승인 이후 발생하는 변경 사항에만 적용된다.
- **소급 적용 배제:** 이미 main에 존재하는 Core 코드, 승격 완료된 PRD, 기존 테스트 상태는 본 규칙의 소급 적용 대상에서 제외된다.

## 3. 핵심 품질 계약 (Quality Contracts)

### 3.1 최소 테스트 존재 요건
- Core 영역(`src/core/*`)에 대한 모든 변경은 반드시 **최소 1개 이상의 테스트**를 포함해야 한다.
- 테스트가 존재하지 않는 변경은 Core 승격이 불가능하다.

### 3.2 테스트 보호 계약
- **수정/제거 규칙:** Core 테스트를 수정하거나 제거하는 경우, 변경 사유가 명확히 기록되어야 한다.
- **감소 제한:** Core 영역에서 테스트 수가 감소하는 변경은 기본적으로 HOLD 상태로 간주하며, 명시적 사유 기록 및 승인이 필수적이다.

## 4. 승인 및 거버넌스 절차 (Governance Flow)

### 4.1 Green ≠ Merge 원칙
- 테스트 PASS는 병합의 필요조건일 뿐, 충분조건이 아니다.
- Core 승격을 위해서는 반드시 다음이 모두 충족되어야 한다:
  1. 테스트 PASS
  2. Reviewer 명시적 판정 (APPROVED)
  3. 승인 기록 존재

### 4.2 변경 유형별 승인 요건
| 변경 유형 | 테스트 요건 | 승인 요건 |
| :--- | :--- | :--- |
| Core 신규 진입 | 1개 이상 필수 | Reviewer APPROVED |
| Core 코드 수정 | 1개 이상 필수 | Reviewer APPROVED |
| Core 테스트 수정 | 사유 기록 필수 | Reviewer APPROVED |
| Core 테스트 제거 | 사유 기록 필수 | Reviewer APPROVED |

## 5. 제약 사항 (Constraints)
- **적용 범위 제한:** 본 규칙은 Core 영역을 보호하기 위한 최소 장치이며, Core 외부 영역에는 동일 강도로 적용되지 않는다.
- **자동화 비포함:** 본 Contract는 자동화 스크립트 추가나 CI 구조 변경을 강제하지 않는다.
