# C-001 : Intent → Contract Map
(PRD-001 : 목차 항목 생성 및 저장)

---

## 1. 목적

본 문서는 PRD-001에 정의된 인간의 의도를
기술적 계약(Contract Spec)으로 변환하기 위한 연결 지도이다.

이 문서에 정의되지 않은 의도는
Contract 및 코드에 존재할 수 없다.

---

## 2. Intent 목록 (PRD 기준)

### A-1. 목차 항목을 생성(Create)하여 DB에 저장한다

- 출처 PRD: 001_목차생성및저장.md
- 의도 요약:
  사용자가 입력한 목차 항목을
  시스템이 데이터베이스에 저장하고,
  고유 ID를 통해 식별 가능하게 만든다.

---

## 3. Intent → Contract 규칙 매핑

### Intent A-1 → Contract Set B-001

| Contract ID | 계약 규칙 요약 | 필수 여부 | 검증 방식 |
|------------|----------------|-----------|-----------|
| B-001-1 | 목차 항목 생성 입력 인터페이스는 필수 필드를 검증해야 한다 | 필수 | 입력 검증 테스트 |
| B-001-2 | 목차 항목은 저장 시 고유 ID를 반드시 부여받아야 한다 | 필수 | 저장 후 응답 검증 |
| B-001-3 | 저장 성공 응답은 생성된 항목의 ID를 포함해야 한다 | 필수 | API 응답 테스트 |
| B-001-4 | 저장된 목차 항목은 ID 기준으로 즉시 조회 가능해야 한다 | 필수 | Create → Read 테스트 |
| B-001-5 | 저장 실패 시, 항목은 DB에 부분 저장되어서는 안 된다 | 필수 | 트랜잭션 테스트 |

---

## 4. Contract 범위 선언

본 Intent(A-1)에 의해 생성되는 Contract는 다음 범위로 제한된다.

포함:
- 목차 항목 생성(Create)
- 저장 결과 확인(Read by ID)
- 필수 필드 검증
- 고유 ID 발급 규칙

제외:
- 목차 항목 수정(Update)
- 목차 항목 삭제(Delete)
- 계층 정렬 로직
- md 파일 연동
- 사용자/권한 개념

---

## 5. 검증 가능성 선언 (Review Gate)

다음 질문에 모두 YES로 답할 수 있어야
PRD-001은 Contract로서 충족된 것으로 판단한다.

- 모든 PRD 문장이 하나 이상의 Contract 규칙에 매핑되었는가?
- 각 Contract 규칙은 YES/NO로 판정 가능한가?
- 저장 성공과 실패가 기술적으로 명확히 구분되는가?
