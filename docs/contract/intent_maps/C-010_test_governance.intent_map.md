# C-010: Core Test Governance (Intent Map)

## 1. 개요 (Intent)
PRD-010의 핵심 의도는 Core 영역에 대한 최소 테스트 존재 요건과 승인 흐름을 명문화하여 고정하는 것이다. 이는 미래의 Core 변경에 대해 테스트 없는 진입을 차단하는 경계선 선언이다.

## 2. Intent-Rule-Evidence 매핑

| Intent (의도) | Rule (구현 규칙) | Evidence (검증 증거) |
| :--- | :--- | :--- |
| **최소 테스트 존재 보장** | Core 영역 변경 시 최소 1개 이상의 테스트가 포함되어야 한다. | PR 변경 사항 내 테스트 존재 확인 |
| **명시적 승인 흐름 고정** | Reviewer의 APPROVED 판정 없이는 Core 병합이 불가능하다. | 승인 기록 (APPROVED) |
| **소급 적용 배제** | 기존 main 코드나 과거 상태에 대해서는 책임을 묻지 않는다. | 신규/수정 PR 대상 여부 확인 |
| **테스트 감소 제어** | 테스트 수 감소 시 사유 기록과 승인 절차를 거친다. | 변경 사유 기록 및 Reviewer 판정 |

## 3. Reviewer 체크리스트 (Human Enforcement)

- [ ] **Presence:** Core 코드 변경에 대해 최소 1개 이상의 테스트가 존재하는가?
- [ ] **Approval:** Reviewer가 명시적으로 APPROVED 판정을 내렸는가?
- [ ] **Reasoning:** 테스트 수정 또는 감소 시 명확한 사유가 기록되었는가?

## 4. 시나리오별 처리 프로세스

### 4.1 테스트 없는 Core 변경
- **상황:** Core 코드를 수정했으나 테스트가 포함되지 않음.
- **결과:** Core 승격 불가능.
- **조치:** 테스트 추가 후 재검토.

### 4.2 테스트 수 감소
- **상황:** Core 테스트 개수가 줄어드는 변경.
- **결과:** HOLD 상태로 간주.
- **조치:** 사유 기록 확인 및 Reviewer 승인 시 허용.
